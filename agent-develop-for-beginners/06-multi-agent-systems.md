# 第六章：多智能体协作（Multi-Agent Systems）

单打独斗的时代结束了。当你试图让一个单一的、装备了无数工具的“全能 Agent”去完成极度复杂的软件开发生命周期的任务时，它往往会崩溃：写完了代码忘了写测试，写了测试又忘了更新 README 文档。

不仅对人类而言术业有专攻，对大模型更是如此——你把它角色设定得越具体（专家化），它在单一任务上的涌现性能就越强。这就引出了当前最激动人心的领域：**（Multi-Agent Systems，简称 MAS）**。

## 1. 为什么需要多智能体协作？

在企业真实的环境中，工作任务不会由老板一个人全包，而是依靠各个部门的高效流转。
同样的道理，在一个 MAS 架构中：

- 有一个 **规划者 Agent (Planner)**：负责拆解需求，把它分解为子任务分发下去。
- 有一个 **搜索专家 Agent (Researcher)**：专门调用 Google 搜索和维基百科获取实时资料。
- 有一个 **代码专家 Agent (Coder)**：接到资料开始撰写代码。哪怕它的工具库中只有 IDE 操作权限。
- 有一个 **质检员 Agent (Reviewer/Critic)**：检查前面输出的代码寻找语法漏洞和逻辑错误。

这样做有两个巨大的好处：
1. **隔离与专注**：每种 Agent 的 Prompt 可以写到极尽专精。它们拥有的工具也被极度收缩在专属领域内，这使得出错和模型产生幻觉的几率大幅降低。
2. **多模型组合，降低金钱成本**：有些简单的总结节点你完全可以用便宜的低参数量模型，仅仅把最贵的最智能的大模型留给 Planner 节点。

## 2. 协作模式：层级、图结构与群聊

Agent 与 Agent 之间是如何沟通协作的呢？常见的架构有以下几种：

### (a). 层级化 (Hierarchical Pattern)
最类似于企业管理模式：也就是前不久很火的 `Manager-Worker`（经理与员工）模式。
你有一个主要的 Agent 作为入口节点路由，它接管用户输入，再使用自己的工具（如：下发任务给 A，下发任务给 B）去驱动子节点工作。它来负责汇总最终结果。

这类框架的代表选手是被广泛喜爱的轻量级框架 **CrewAI**。

> **📍 典型场景**：构建一个"调研 + 写作 + 审核"的内容生产流水线。Planner 接单后，依次派发给 Researcher 查资料 → Writer 撰写正文 → Editor 审核润色，结果层层上交汇总。流程固定、分工清晰，层级化最为合适。

### (b). 网状或图结构协作 (Network/Graph Routing)
这类协作没有固定的“高低级”之分。比如由 **LangGraph** 所编排的网络，A 办完事，根据特定状态把数据发给了 B，B 报错了，状态流又抛回给了 A 进行自我修复。这在上一章的介绍中表现得淋漓尽致。

> **📍 典型场景**：构建一个"自动写代码 → 运行测试 → 修复报错 → 重新测试"的 CI/CD 自动化 Agent。因为测试可能需要多次回退重写，必须支持任意节点间的状态流转和条件分叉，图结构是唯一合适的选择。

### (c). 会话群聊 (Chat/Dialogue Based MAS)
这就是微软开源的极富创造力的框架 **AutoGen** 诞生的本意。
在这个模式下，Agent 就像是在一个会议群（Group Chat）里说话。
Speaker 讲完一段话后，系统会用特定的 LLM 发言权选举算法决定下一个该由谁（比如测试员、产品经理、审核员）对上一句话进行点评或补全。这也是人类开会最自然的映射。在开源框架 **OWL** 的设计中，其 Planner 利用不同的多模态能力也是从侧面打通了这种交流体系。

> **📍 典型场景**：构建一个"产品迭代讨论小组"——产品经理 Agent 提出需求、工程师 Agent 评估可行性并反对某个功能、设计师 Agent 给出交互建议——通过多轮自由对话最终达成一致决策，而非按固定顺序依次执行。

## 3. 如何选择适合你的协作模式？

| 维度 | 层级化（CrewAI） | 图结构（LangGraph） | 群聊（AutoGen） |
|------|:-----------:|:------------:|:-----------:|
| **流程可控程度** | ⭐⭐⭐ 高（固定顺序） | ⭐⭐⭐⭐⭐ 最高（精细控制） | ⭐⭐ 低（自由发散） |
| **上手难度** | 🟢 低 | 🔴 高 | 🟡 中 |
| **适合任务类型** | 流程固定的线性生产任务 | 含循环/错误回退/条件分叉的复杂任务 | 开放探索性/创意讨论类任务 |
| **典型使用场景** | 内容生产、报告生成 | 代码自动测试修复、DevOps 自动化 | 头脑风暴、设计评审、辩论 |
| **调试难度** | 🟢 容易（流程清晰） | 🟡 中等（看日志+图表） | 🔴 困难（对话不可控） |

> 💡 **选型建议**：对于大多数业务场景，推荐优先尝试 **CrewAI（层级化）** 快速验证思路；当你发现需要处理错误回退、条件路由或人工审核断点时，再迁移到 **LangGraph**；而 **AutoGen 的群聊模式**则更适合用于探索实验或需要多视角互相质疑的决策推演场景。

## 4. 在 Python 中的简单多智能体通信机制

如果你不想被繁重的框架绑架，你其实也可以通过纯 Python 实现最简单的多智能体交互逻辑。**归根结底，前一个 Agent 的输出（Output），就是下一个 Agent 上下文输入历史（History）的一部分。**

```python
# 伪代码：最简单的硬编码协作流
def multi_agent_workflow(idea):
    # 第一步：交给文案小王（Writer Agent）
    writer_prompt = f"你是一流的科普作家，请将这个想法：{idea} 扩写为千字初稿。"
    draft = call_llm(context=[writer_prompt]) # -> 获得上千字初稿

    # 第二步：交给审核员老李（Reviewer Agent）
    reviewer_prompt = f"你严格的文字编辑，请审视并严厉批判以下草稿漏洞：\n{draft}"
    feedback = call_llm(context=[reviewer_prompt]) # -> 获得修改意见

    # 第三步：文案小王看着意见再改（Revisor Agent）
    revisor_prompt = f"这是你的原稿：{draft}\n这是批评意见：{feedback}\n请输出最终完美版本。"
    final_article = call_llm(context=[revisor_prompt])

    return final_article
```
这就是多智能体的本质：**让思想的结晶在不同的专业视域中多次碰撞（Multi-Turn Interaction）。**
